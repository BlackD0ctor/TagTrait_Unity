1. 현재 단계 목표
- 기존 소멸 태그에 StackPolicy.Refresh를 적용한다.
- PerishableTagAdd로 전달된 새 duration을 기준으로 기존 TagInstance의 Duration과 RemainingTime을 교체한다.
- 기존 TagInstance 참조와 StackCount를 유지한다.
- Trait 생명주기를 다시 호출하지 않는다.
- OnTagUpdated를 한 번 발생시키고 DurationRefreshed 사유를 전달한다.
- Tick 처리 중 Refresh된 태그는 현재 Tick의 남은 시간 처리를 건너뛰고 다음 독립 Tick부터 감소하도록 한다.
- MaxDuration과 StackCount 정책은 구현하지 않는다.

2. 이전 단계 전제
- 단계 6-1의 PerishableTagAdd 및 duration 정규화가 검증되었다.
- 단계 6-2의 Tick, 만료, isTicking, 스냅샷 순회가 검증되었다.
- 단계 7의 소멸→영구 전환이 검증되었다.
- 전제가 충족되지 않으면 구현을 시작하지 말고 누락 내용을 보고한다.

4. Refresh 적용 조건
- 다음 조건을 모두 만족할 때 적용한다.
-- 같은 TagID의 기존 Instance가 존재
-- 기존 Instance.Definition과 요청 Definition이 같은 참조
-- 기존 Instance.IsPerishable == true
-- Definition.StackPolicy == StackPolicy.Refresh
- 기존 태그가 없으면 단계 6-1의 신규 소멸 태그 추가 경로를 사용한다.
- 같은 ID의 다른 Definition은 Error 로그 후 false이다.

5. 정책별 중복 결과
- 기존 소멸 + Refresh:
-- Refresh 성공
-- true
-- OnTagUpdated 한 번
- 기존 소멸 + None:
-- false
-- 상태 변화 없음
-- 이벤트 없음
- 기존 소멸 + MaxDuration:
-- 단계 9 전까지 false
- 기존 소멸 + StackCount:
-- 단계 11 전까지 false
- 기존 영구 태그:
-- false
-- 상태 변화 없음
- TagAdd의 소멸→영구 전환은 단계 7 정책을 유지한다.

6. duration 처리
- PerishableTagAdd 호출 초기에 단계 6-1의 동일 검증과 정규화를 수행한다.
- 유한한 양수만 허용한다.
- 0, 음수, NaN, 양·음 Infinity는 Error와 false이다.
- 소수점 첫째 자리, MidpointRounding.AwayFromZero를 유지한다.
- 작은 양수는 최소 0.1f로 보정한다.
- Refresh에서도 duration은 호출당 한 번만 정규화한다.
- Tick 감소값과 RemainingTime을 반복 정규화하지 않는다.

7. Refresh 시간 교체 규칙
- 정규화된 새 duration으로 다음 값을 모두 교체한다.
-- Duration = normalizedDuration
-- RemainingTime = normalizedDuration
- 현재 RemainingTime보다 긴지 짧은지 비교하지 않는다.
- 더 짧은 duration도 무조건 적용한다.
- 더 긴 duration도 무조건 적용한다.
- 일반적으로는 원래 기준시간으로 남은 시간을 되돌리는 용도로 사용할 수 있으나, API 계약은 전달된 새 duration으로 완전 갱신하는 것이다.

8. 동일 값 Refresh
- 정규화된 새 Duration과 RemainingTime이 기존 값과 같더라도 성공으로 처리한다.
- true를 반환한다.
- OnTagUpdated를 한 번 발생시킨다.
- Reason은 DurationRefreshed이다.
- 수치 변화 여부가 아니라 Refresh 행위 성공 여부를 기준으로 한다.
- Source가 달라질 수 있으므로 동일 값 Refresh를 무시하지 않는다.

9. TagInstance 내부 API
- 다음 의미의 internal 메서드를 추가한다.
-- internal bool TryRefreshDuration(float duration)
- IsPerishable == false이면 false이다.
- IsPerishable == true이면:
-- Duration = duration
-- RemainingTime = duration
-- StackCount 유지
-- true
- 같은 값이어도 true이다.
- 전달 duration은 이미 검증·정규화된 값이다.
- 이 메서드는 정규화, 로그, Trait, 이벤트, Dictionary, 재진입 처리를 하지 않는다.

10. 참조 및 StackCount
- 새 TagInstance를 만들지 않는다.
- Dictionary의 Instance를 교체하지 않는다.
- 기존 TagInstance 참조를 유지한다.
- 기존 TagScan 얕은 스냅샷과 외부 참조에서 갱신된 상태가 보인다.
- StackCount는 변경하지 않는다.
- 스택 증가, 감소, 초기화 모두 수행하지 않는다.

11. Trait 생명주기
- Refresh 시 다음을 호출하지 않는다.
-- OnAdd
-- OnRemove
-- OnActivate
- 태그 보유 여부는 변하지 않고 시간 상태만 변경된다.
- 최초 추가 시 호출된 OnAdd를 다시 호출하지 않는다.

12. Updated 이벤트
- 성공한 Refresh에서 OnTagUpdated를 정확히 한 번 발생시킨다.
- 이벤트 데이터:
-- Container = 현재 TagContainer
-- Definition = 기존 Instance.Definition
-- Instance = 갱신된 동일 Instance
-- Source = PerishableTagAdd의 source
-- Reason = TagChangeReason.DurationRefreshed
-- PreviousDuration = 갱신 직전 Duration
-- PreviousRemainingTime = 갱신 직전 RemainingTime
-- PreviousStackCount = 갱신 직전 StackCount
- 이벤트 콜백에서는 이미 새 Duration과 RemainingTime이 보여야 한다.
- OnTagAdded와 OnTagRemoved는 발생시키지 않는다.
- 구독자 예외 정책은 기존 SafeInvoke를 유지한다.
-- 갱신 상태 유지
-- 다음 구독자 계속 호출
-- 예외 비전파
-- API true

13. PerishableTagAdd 처리 순서
- Definition 유효성 검사
- duration 유효성 검사 및 정규화
- 동일 TagID 재진입 가드 획득
- 기존 Instance 조회
- 기존 태그가 없으면 신규 소멸 추가 경로
- 기존 태그가 있으면 Definition 참조 비교
- 기존 영구 태그이면 false
- StackPolicy 확인
- Refresh가 아니면 현재 단계 정책에 따라 false
- Previous 값 보관
- TryRefreshDuration 호출
- Tick 처리 중이면 tickModifiedTagIDs에 ID 기록
- OnTagUpdated 발생
- finally에서 processingTagIDs 해제
- true

14. 재진입
- Refresh 시작부터 Updated 이벤트 종료까지 processingTagIDs를 유지한다.
- Updated 콜백에서 같은 ID의 TagAdd, TagSub, PerishableTagAdd, TagActivate는 false이다.
- 같은 ID 재진입에는 상태 변화와 추가 이벤트가 없다.
- 다른 ID 변경과 조회 API는 허용한다.
- finally에서 가드를 해제한다.

15. Tick 도중 Refresh 정책
- Refresh 성공을 우선한다.
- Refresh를 무시하고 기존 만료를 강제하지 않는다.
- Tick 처리 중 다른 TagID 콜백에서 Refresh가 성공하면 새 시간값을 즉시 적용한다.
- 해당 ID를 tickModifiedTagIDs에 기록한다.
- 해당 태그의 현재 Tick 처리 차례가 아직 남아 있으면 건너뛴다.
- 다음 독립 Tick부터 새 RemainingTime을 감소시킨다.
- 이미 현재 Tick에서 먼저 감소한 뒤 다른 ID 콜백으로 Refresh되었다면 Refresh된 전체 시간이 최종 상태이다.
- 이미 제거된 뒤 PerishableTagAdd가 호출되면 새 Instance 추가 경로를 사용하며 현재 Tick 스냅샷에는 없으므로 다음 Tick부터 감소한다.
- 태그 자신의 만료 처리 중 같은 ID Refresh는 processingTagIDs에 의해 차단되며 기존 만료를 계속한다.
- 강제 소멸 정책은 Refresh에 섞지 않는다.

16. tickModifiedTagIDs
- TagContainer에 다음 private 필드를 추가한다.
-- private readonly HashSet<TagID> tickModifiedTagIDs = new HashSet<TagID>();
- public 또는 internal로 노출하지 않는다.
- Tick 시작 시 Clear한다.
- isTicking == true인 동안 Refresh 성공 시 해당 ID를 Add한다.
- Tick 스냅샷 항목 처리 전에 해당 ID가 포함되어 있으면 건너뛴다.
- Tick 종료의 finally에서 Clear한다.
- isTicking도 기존 finally에서 false로 해제한다.
- 예상하지 못한 종료 이후 다음 Tick에 이전 변경 기록이 남지 않아야 한다.
- 새 태그 추가는 스냅샷에 없으므로 별도 기록이 필요 없다.
- 제거된 태그는 기존 존재·참조 확인으로 건너뛴다.

17. Tick 처리 순서 보강
- 각 스냅샷 Instance 처리 전 다음 순서를 권장한다.
-- 현재 Dictionary에 같은 ID가 존재하는지
-- 현재 Instance가 스냅샷 Instance와 같은 참조인지
-- tickModifiedTagIDs에 해당 ID가 있는지
-- 여전히 IsPerishable인지
-- processingTagIDs 가드 획득
-- 시간 감소 및 만료 처리
- tickModifiedTagIDs에 있으면 해당 항목은 현재 Tick에서 완전히 건너뛴다.
- 처리 순서는 공개 계약으로 보장하지 않는다.
- 순서와 관계없이 Refresh된 최종 태그는 새 전체 시간으로 Tick을 종료해야 한다.

18. 실패 정책과 로그

- duration 오류: Error + false
- 같은 ID 다른 Definition: Error + false
- 기존 영구 태그: 로그 없이 false

- 미구현 MaxDuration: 로그 없이 false

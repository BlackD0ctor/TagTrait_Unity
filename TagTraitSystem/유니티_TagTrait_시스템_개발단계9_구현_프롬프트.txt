1. 현재 단계 목표
- 기존 소멸 태그에 StackPolicy.MaxDuration을 적용한다.
- 기존 기준 Duration과 새 duration 중 큰 값을 Duration으로 유지한다.
- 기존 남은 시간과 새 duration 중 큰 값을 RemainingTime으로 적용한다.
- Tick 처리 중에는 Tick 시작 시점의 RemainingTime을 비교 기준으로 사용하여 처리 순서 의존성을 제거한다.
- 실제 Duration 또는 RemainingTime이 증가한 경우에만 성공 처리한다.
- 성공 시 기존 TagInstance를 유지하고 OnTagUpdated를 한 번 발생시킨다.
- 실제 연장이 없으면 false, 이벤트 없음, tickModifiedTagIDs 기록 없음으로 처리한다.
- StackCount 정책은 구현하지 않는다.

2. 이전 단계 전제
- 단계 6-1의 PerishableTagAdd와 duration 정규화가 검증되었다.
- 단계 6-2의 Tick, 만료, isTicking, Tick 스냅샷이 검증되었다.
- 단계 7의 소멸→영구 전환이 검증되었다.
- 단계 8의 Refresh와 tickModifiedTagIDs가 검증되었다.
- 전제가 충족되지 않으면 구현하지 말고 누락 내용을 보고한다.

4. MaxDuration 적용 조건
- 다음 조건을 모두 만족할 때만 적용 대상으로 본다.
-- 같은 TagID의 기존 Instance가 존재
-- 기존 Instance.Definition과 요청 Definition이 같은 참조
-- 기존 Instance.IsPerishable == true
-- Definition.StackPolicy == StackPolicy.MaxDuration
- 기존 태그가 없으면 단계 6-1의 신규 소멸 태그 추가 경로를 사용한다.
- 기존 영구 태그에는 적용하지 않는다.
- 같은 ID의 다른 Definition은 Error 로그 후 false이다.

5. 정책별 기존 소멸 태그 중복 결과
- StackPolicy.None:
-- false
-- 상태 변화 없음
-- 이벤트 없음
- StackPolicy.Refresh:
-- 단계 8 Refresh 정책 적용
- StackPolicy.MaxDuration:
-- 현재 단계의 최대시간 연장 정책 적용
- StackPolicy.StackCount:
-- 단계 11 전까지 false
- TagAdd의 소멸→영구 전환은 단계 7 정책을 유지한다.

6. duration 검증 및 정규화
- 단계 6-1과 동일한 검증 및 정규화를 사용한다.
- 유한한 양수만 허용한다.
- 0, 음수, NaN, PositiveInfinity, NegativeInfinity는 Error와 false이다.
- 소수점 첫째 자리, MidpointRounding.AwayFromZero를 유지한다.
- 작은 양수는 최소 0.1f로 보정한다.
- 정규화는 PerishableTagAdd 호출 시 한 번만 수행한다.
- Tick의 deltaTime과 감소 후 RemainingTime은 정규화하지 않는다.

7. MaxDuration 핵심 계산식
- Tick 외부에서 호출될 때:
-- comparisonRemainingTime = instance.RemainingTime
- Tick 처리 중 호출될 때:
-- 해당 TagID의 Tick 시작 RemainingTime 기록이 있으면 comparisonRemainingTime = tickStartRemainingTimes[tagID]
-- 기록이 없으면 comparisonRemainingTime = instance.RemainingTime
- 계산:
-- newDuration = max(instance.Duration, normalizedDuration)
-- newRemainingTime = max(comparisonRemainingTime, normalizedDuration)
- B안처럼 newRemainingTime을 newDuration으로 무조건 복원하지 않는다.
- 과거의 가장 긴 Duration 전체를 약한 재적용으로 다시 시작시키지 않는다.

8. 의미 예시
- 기존 Duration = 90초
- 기존 RemainingTime = 20초
- 새 duration = 30초
- 결과:
-- Duration = 90초
-- RemainingTime = 30초
- 다음 결과는 금지한다.
-- RemainingTime = 90초
- 기존 Duration = 90초, RemainingTime = 40초, 새 duration = 30초이면 실제 연장이 없으므로 no-op이다.
- 기존 Duration = 90초, RemainingTime = 20초, 새 duration = 120초이면 Duration과 RemainingTime 모두 120초가 된다.

9. 실제 연장 여부
- 다음 중 하나라도 실제 증가할 때만 성공 처리한다.
-- newDuration > instance.Duration + TimeEpsilon
-- newRemainingTime > comparisonRemainingTime + TimeEpsilon
- 둘 다 false이면:
-- 상태 변경 없음
-- false 반환
-- OnTagUpdated 없음
-- tickModifiedTagIDs 기록 없음
-- Trait 호출 없음
-- 로그 없음
- MaxDuration은 실제 연장 행위를 의미하므로 Refresh와 달리 동일 값 재적용을 성공 처리하지 않는다.

10. Tick 처리 후 호출 시 감소값 복원 금지
- Tick 처리 중 해당 태그가 이미 감소한 뒤 MaxDuration이 호출될 수 있다.
- 실제 연장이 없는 경우 현재 감소 결과를 Tick 시작값으로 되돌리지 않는다.
- 예:
-- Tick 시작 RemainingTime = 20초
-- Tick 처리 후 현재 RemainingTime = 19초
-- 새 duration = 10초
-- comparisonRemainingTime = 20초
-- newRemainingTime = 20초이지만 실제 연장 조건은 Tick 시작 기준으로 false
-- 상태를 적용하지 않고 현재 RemainingTime = 19초를 유지한다.
- 실제 연장이 있는 경우에만 계산된 newDuration과 newRemainingTime을 저장한다.

11. TagInstance 내부 API
- 다음 의미의 internal 메서드를 추가한다.
-- internal bool TryExtendDurationToMax(float duration, float comparisonRemainingTime, float epsilon)
- 또는 동일 책임의 명확한 internal 메서드를 사용한다.
- 메서드 책임:
-- IsPerishable이 false이면 false
-- newDuration = max(Duration, duration)
-- newRemainingTime = max(comparisonRemainingTime, duration)
-- epsilon 기준 실제 증가 여부 판정
-- 실제 증가가 없으면 어떤 필드도 변경하지 않고 false
-- 실제 증가가 있으면 Duration과 RemainingTime을 계산값으로 설정하고 true
-- StackCount 유지
- 이 메서드는 다음을 수행하지 않는다.
-- duration 검증 또는 정규화
-- Trait 호출
-- 이벤트
-- 로그
-- Dictionary 변경
-- 재진입 처리
-- tickModifiedTagIDs 처리
- epsilon을 TagInstance 내부에 별도 상수로 중복 선언하지 않는다.

12. Tick 시작 RemainingTime 추적
- TagContainer에 다음 private 필드를 추가한다.
-- private readonly Dictionary<TagID, float> tickStartRemainingTimes = new Dictionary<TagID, float>();
- public 또는 internal로 노출하지 않는다.
- Tick 시작 시 스냅샷을 구성하면서 각 소멸 태그의 TagID와 시작 RemainingTime을 기록한다.
- 현재 Tick 시작 시점에 존재한 실제 Instance만 기록한다.
- Tick 종료의 finally에서 Clear한다.
- 예상하지 못한 종료 후 다음 Tick에 기록이 남지 않아야 한다.
- 새로 추가된 태그는 Tick 시작 기록이 없으므로 현재 Instance.RemainingTime을 비교 기준으로 사용한다.

13. Tick 시작값 기록 시점
- 잘못된 deltaTime 검증이 끝난 뒤 기록한다.
- isTicking을 true로 설정한 뒤 try 내부에서 Tick 스냅샷과 함께 기록한다.
- 기록값은 해당 Tick에서 태그가 감소하기 전의 RemainingTime이다.
- Tick 중 Refresh 또는 MaxDuration으로 상태가 변경되어도 기록값은 현재 Tick 동안 바꾸지 않는다.
- 기록은 처리 순서 독립성을 위한 비교 기준이며 외부 공개 상태가 아니다.

14. Tick 처리 순서 독립성
- Tick 시작 RemainingTime = 20초, deltaTime = 1초, 새 duration = 30초인 경우:
-- 태그 처리 전 MaxDuration: 30초로 연장, tickModifiedTagIDs 기록, 현재 Tick 건너뜀, 최종 30초
-- 태그 처리 후 MaxDuration: 현재 19초이지만 시작값 20초와 30초를 비교하여 30초로 연장, 최종 30초
- 새 duration = 10초인 경우:
-- 처리 전: 실제 연장 없음, tickModified 미기록, 정상 Tick 감소, 최종 19초
-- 처리 후: 실제 연장 없음, 현재 감소 결과 19초 유지
- 같은 입력은 처리 순서와 관계없이 같은 최종 상태를 만들어야 한다.

15. 기존 Instance와 StackCount
- 새 TagInstance를 만들지 않는다.
- Dictionary Instance를 교체하지 않는다.
- Definition 참조와 IsPerishable 상태를 유지한다.
- StackCount를 변경하지 않는다.
- 스택 증가, 감소, 초기화는 수행하지 않는다.
- 기존 TagScan 얕은 스냅샷과 외부 참조에서 연장된 상태가 보여야 한다.

16. Trait 생명주기
- MaxDuration 적용 시 다음을 호출하지 않는다.
-- OnAdd
-- OnRemove
-- OnActivate
- 태그 보유 상태는 변하지 않고 시간 상태만 변경된다.

17. Updated 이벤트
- 실제 연장에 성공한 경우에만 OnTagUpdated를 정확히 한 번 발생시킨다.
- Reason = TagChangeReason.DurationExtendedToMax
- 이벤트 데이터:
-- Container = 현재 TagContainer
-- Definition = 기존 Instance.Definition
-- Instance = 연장된 동일 Instance
-- Source = PerishableTagAdd의 source
-- PreviousDuration = 공개 API 호출 직전 실제 Duration
-- PreviousRemainingTime = 공개 API 호출 직전 실제 RemainingTime
-- PreviousStackCount = 공개 API 호출 직전 StackCount
- Previous 값은 Tick 시작값이 아니라 실제 API 호출 직전 상태이다.
- 이벤트 콜백에서는 적용된 newDuration과 newRemainingTime이 보여야 한다.
- OnTagAdded와 OnTagRemoved는 발생시키지 않는다.
- 구독자 예외 시 기존 SafeInvoke 정책을 유지한다.
-- 상태 유지
-- 다음 구독자 계속 호출
-- 예외 비전파
-- API true

18. PerishableTagAdd 처리 순서
- Definition 검증
- duration 검증 및 정규화
- 동일 TagID processingTagIDs 가드 획득
- 기존 Instance 조회
- 기존 태그가 없으면 신규 소멸 추가 경로
- 기존 태그가 있으면 Definition 참조 검증
- 기존 영구 태그이면 false
- StackPolicy 분기
- MaxDuration이면 comparisonRemainingTime 결정
- API 호출 직전 Previous 값 보관
- TryExtendDurationToMax 호출
- false이면 이벤트 없이 false
- true이고 isTicking이면 tickModifiedTagIDs에 ID 기록
- OnTagUpdated 발생
- finally에서 processingTagIDs 해제
- true 반환

19. tickModifiedTagIDs 연동
- 실제 연장에 성공한 경우에만 기록한다.
- 실제 연장이 없으면 기록하지 않는다.
- 성공 후 현재 Tick에서 해당 태그의 처리 차례가 남아 있으면 건너뛴다.
- 이미 처리된 뒤 연장되었으면 계산된 연장 상태가 최종 상태이다.
- 다음 독립 Tick부터 감소한다.
- 단계 8의 Tick 시작 및 finally Clear 정책을 유지한다.

20. 자기 만료 처리와 재진입
- 자신의 만료 처리 중 같은 ID에 PerishableTagAdd를 호출하면 processingTagIDs로 차단한다.
- MaxDuration으로 자기 자신을 되살리지 않는다.
- 다른 ID의 콜백에서 아직 자기 만료 파이프라인이 시작되지 않은 태그에는 적용할 수 있다.
- Updated 이벤트 종료까지 동일 ID 가드를 유지한다.
- 다른 ID 변경과 조회 API는 허용한다.

21. 실패 및 로그 정책
- Definition 오류: Error + false
- duration 오류: Error + false
- 같은 ID 다른 Definition: Error + false
- 기존 영구 태그: 로그 없이 false
- StackPolicy.None: 로그 없이 false
- StackPolicy.StackCount: 단계 11 전까지 로그 없이 false
- 실제 연장 없음: 로그 없이 false
- 동일 ID 재진입: 로그 없이 false
- 모든 실패에서 Trait과 변경 이벤트가 없어야 한다.
- Error 로그는 영어이며 단계 12 TagDiagnostics 교체 한글 주석을 유지한다.

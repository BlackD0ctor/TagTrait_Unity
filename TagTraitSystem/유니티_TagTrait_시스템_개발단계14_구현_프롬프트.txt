1. 현재 단계 목표
- TagTrait 시스템 초기 기능 버전을 0.1.0으로 정리한다.
- 새로운 기능을 추가하지 않는다.
- 공개 API, enum 숫자값, 직렬화 계약을 0.1.0 기준으로 문서화한다.
- 기능 변화가 없는 안전한 코드 정리만 수행한다.
- 오래된 단계 주석과 직접 Debug 로그 잔존 여부를 정리한다.
- Runtime, Editor, Samples, Tests Assembly 참조 방향을 최종 점검한다.
- 전체 사용법, 공개 API, 알려진 제한사항, 변경 기록, 버전, 최종 검증 체크리스트를 작성한다.
- 사용자가 Unity 컴파일, 전체 EditMode 테스트, Validator, Sample, Windows IL2CPP Development Build를 검증할 수 있도록 최종 체크리스트를 제공한다.
- 사용자가 실제 검증을 완료하기 전에는 초기 버전 완료로 판정하지 않는다.

5. enum 숫자값 동결
- 기존 값을 변경하지 않는다.
- TagID:
-- None = 0
-- TestAlpha = 1
-- TestBeta = 2
-- TestGamma = 3
-- SampleKeyword = 100
-- SamplePermanent = 101
-- SampleRefresh = 102
-- SampleMaxDuration = 103
-- SampleStackCount = 104
- TagCategory:
-- None = 0
-- Keyword = 1
-- Status = 2
- StackPolicy:
-- None = 0
-- MaxDuration = 1
-- Refresh = 2
-- StackCount = 3
- TagChangeReason:
-- Added = 0
-- Removed = 1
-- DurationRefreshed = 2
-- DurationExtendedToMax = 3
-- StackIncreased = 4
-- StackDecreased = 5
-- ChangedToPermanent = 6
-- Expired = 7
- ExclusiveGroup:
-- None = 0
- 문서에 숫자값 변경 시 직렬화 마이그레이션이 필요하다고 명시한다.

6. Test TagID 유지
- TestAlpha, TestBeta, TestGamma를 제거하지 않는다.
- 숫자값 1, 2, 3을 변경하지 않는다.
- 영어 주석 또는 문서에 다음 의미를 유지한다.
-- Development and test placeholders.
-- Production projects should add project-specific TagID values.
-- Do not reuse serialized numeric values without a migration plan.
- 테스트용 ID가 실제 프로젝트 ID 체계가 아님을 문서화한다.

7. Sample TagID 유지
- SampleKeyword~SampleStackCount를 유지한다.
- 숫자값 100~104를 변경하지 않는다.
- 각 Sample 멤버의 영어 주석을 유지한다.
-- Sample-only TagID. Replace or extend it with project-specific IDs as needed.
- 실제 프로젝트에서는 프로젝트별 TagID를 추가하거나 Sample ID를 교체할 수 있다고 문서화한다.
- 기존 Sample 에셋이 존재한다면 숫자값 변경 전 마이그레이션이 필요함을 안내한다.

8. 직렬화 계약 문서화
- TagDefinition의 실제 serialized field 이름과 기본값을 코드 기준으로 문서화한다.
- 최소 포함:
-- tagID
-- tagName
-- category
-- stackPolicy
-- trait
-- maxStackCount
-- exclusiveGroup
-- priority
-- isSaveable
- 실제 코드의 정확한 이름과 기본값을 확인한다.
- TagInstance Runtime 상태:
-- Definition
-- Duration
-- RemainingTime
-- StackCount
-- IsPerishable
- enum 숫자값과 serialized field 이름을 임의로 변경하지 않는다.
- 필드 이름 변경에는 FormerlySerializedAs 또는 별도 마이그레이션 설계가 필요하다고 명시한다.

22. 성능 정적 점검
- 벤치마크를 새로 작성하지 않는다.
- 프로파일링 결과 없이 최적화하지 않는다.

-- 다중 StackCount Tick에서 스택 수 비례 반복문 없음
-- TagScan 배열 스냅샷
-- Tick 스냅샷
-- 이벤트 구독자 복사 또는 할당
-- Editor Validator 할당은 Editor 전용
-- Sample이 Runtime 핵심 경로에 영향 없음
- 저할당 public API를 추가하지 않는다.
- 실제 성능 문제가 확인되면 별도 설계를 권장한다.

23. 테스트 정책

- 코드 정리로 실제 동작이 변한 경우에만 최소 회귀 테스트를 추가한다.
- 기존 테스트를 삭제하거나 완화하지 않는다.
- 테스트 실행 결과는 사용자가 Unity에서 확인해야 한다.

-- Existing test suite not executed.
-- Manual Unity verification required.
- 사용자가 실제 결과를 전달한 경우에만 해당 상태를 갱신한다.

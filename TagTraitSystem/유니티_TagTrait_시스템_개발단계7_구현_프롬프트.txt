1. 현재 단계 목표
- 기존 소멸 태그에 TagAdd가 호출되면 동일 TagInstance를 영구 상태로 전환한다.
- 전환 성공 시 OnTagUpdated를 한 번 발생시킨다.
- 전환 사유는 TagChangeReason.ChangedToPermanent를 사용한다.
- 전환 과정에서 OnAdd, OnRemove, OnActivate를 호출하지 않는다.
- 기존 영구 태그에 대한 중복 TagAdd와 PerishableTagAdd 임시 정책은 유지한다.
- 전환된 태그가 Tick의 영향을 받지 않도록 기존 Tick 로직과 연동한다.

2. 이전 단계 전제
- 개발 단계 6-1과 6-2가 사용자의 Unity 검증을 통과한 것으로 전제한다.
- 다음 기능이 존재해야 한다.
-- TagAdd
-- PerishableTagAdd
-- Tick
-- TagChangeEventData
-- OnTagAdded, OnTagRemoved, OnTagUpdated
-- Trait 생명주기와 SafeInvoke
-- processingTagIDs 재진입 가드
-- isTicking 중첩 Tick 가드
- 전제가 충족되지 않으면 구현을 시작하지 말고 누락 내용을 보고한다.

3. 수정 및 생성 파일
- 수정:
-- Assets/TagTraitSystem/Runtime/Core/TagInstance.cs
-- Assets/TagTraitSystem/Runtime/Components/TagContainer.cs
- 생성:
-- Assets/TagTraitSystem/Tests/EditMode/TagContainerPermanentConversionTests.cs
- 기존 테스트 파일을 삭제하거나 임의로 이동하지 않는다.
- 새 공개 타입 또는 공개 API를 추가하지 않는다.

4. 전환 방향
- 허용:
-- 기존 소멸 태그 + TagAdd -> 영구 태그로 전환
- 허용하지 않음:
-- 기존 영구 태그 + PerishableTagAdd -> false
- 단계 7에서는 소멸 -> 영구 단방향 전환만 구현한다.
- 영구 -> 소멸 전환 기능을 추가하지 않는다.

5. TagInstance 내부 전환 API
- 다음 시그니처를 구현한다.
-- internal bool TryConvertToPermanent()
- 이미 영구 상태이면 false를 반환한다.
- 소멸 상태이면 다음 상태로 변경하고 true를 반환한다.
-- Duration = 0f
-- RemainingTime = 0f
-- IsPerishable = false
-- StackCount = 기존 값 유지
-- Definition = 기존 참조 유지
- 새 TagInstance를 생성하거나 Dictionary 값을 교체하지 않는다.
- Trait 호출, 이벤트 발생, 로그 출력, Dictionary 접근을 수행하지 않는다.
- 내부 상태 변경만 담당한다.

6. 동일 TagInstance 유지
- 전환 전후 TagInstance 참조가 동일해야 한다.
- 기존 TagScan 얕은 스냅샷이 같은 객체의 영구 상태 변화를 관찰할 수 있어야 한다.
- 외부에서 보관한 TagInstance 참조가 끊기지 않아야 한다.
- Dictionary에서 기존 Instance를 제거하고 새 Instance를 삽입하는 방식을 사용하지 않는다.

7. StackCount 정책
- 전환 시 StackCount를 변경하지 않는다.
- 1로 강제 초기화하지 않는다.
- 증가 또는 감소시키지 않는다.
- 현재 단계에서는 일반적으로 1이지만 단계 11 이후의 소멸 StackCount와 호환되도록 기존 값을 보존한다.
- 전환 호출의 단일 사유는 ChangedToPermanent이다.
- 같은 호출에서 StackIncreased를 함께 처리하지 않는다.
- 영구 StackCount 중복 추가는 개발 단계 10에서 구현한다.

8. Trait 생명주기
- 전환 시 다음 콜백을 호출하지 않는다.
-- OnAdd
-- OnRemove
-- OnActivate
- 태그의 보유 여부는 유지되고 유한성만 변경되므로 생명주기를 재실행하지 않는다.
- 정상 Keyword도 동일하게 전환할 수 있으나 Trait 호출은 없다.
- Trait 콜백 횟수는 최초 추가 시점에서 증가한 값 그대로 유지되어야 한다.

9. TagAdd 처리 분기
- 기존 Definition 유효성 검사와 동일 TagID 가드 획득 순서를 유지한다.
- 가드 내부에서 기존 TagID의 Instance를 먼저 조회한다.
- 권장 분기 순서:
-- 같은 ID의 기존 Instance가 없음: 기존 영구 태그 신규 추가 흐름
-- 같은 ID의 기존 Instance가 있음: Definition 참조 비교
-- Definition 참조가 다름: Error 로그 후 false
-- 같은 Definition이며 기존 Instance가 소멸 상태: 영구 전환
-- 같은 Definition이며 이미 영구 상태: 기존 중복 정책에 따라 false
- TryAddInstance를 호출하기 전에 기존 Instance 전환 가능성을 처리한다.
- 소멸 태그 전환을 중복 실패로 처리하지 않는다.

10. 전환 처리 순서
- TagAdd Definition 입력 검증
- 동일 TagID processingTagIDs 가드 획득
- 기존 Instance 조회
- 기존 Instance.Definition과 요청 Definition 참조 비교
- 전환 직전 Previous 값 보관
- TryConvertToPermanent 호출
- 성공 시 TagChangeEventData 생성
- OnTagUpdated SafeInvoke
- finally에서 processingTagIDs 가드 해제
- true 반환

11. 전환 이벤트
- 전환 성공 시 OnTagUpdated를 정확히 한 번 발생시킨다.
- 다음 이벤트는 발생시키지 않는다.
-- OnTagAdded
-- OnTagRemoved
- TagChangeEventData 값:
-- Container = 현재 TagContainer
-- Definition = 기존 Instance.Definition
-- Instance = 전환된 동일 TagInstance
-- Source = TagAdd에 전달된 source
-- Reason = TagChangeReason.ChangedToPermanent
-- PreviousDuration = 전환 직전 Duration
-- PreviousRemainingTime = 전환 직전 RemainingTime
-- PreviousStackCount = 전환 직전 StackCount
- 이벤트 콜백에서 현재 Instance 상태는 다음이어야 한다.
-- Duration = 0f
-- RemainingTime = 0f
-- IsPerishable = false
-- StackCount = 전환 전 값
- 이벤트 콜백에서 TagCheck와 TryGetTag는 계속 성공해야 한다.

12. 이벤트 예외 정책
- 기존 SafeInvoke를 사용한다.
- 한 구독자가 예외를 던져도 다음 구독자를 계속 호출한다.
- 전환 상태를 롤백하지 않는다.
- API는 true를 반환한다.
- 예외를 호출자에게 다시 던지지 않는다.
- Error 로그 정책과 단계 12 TagDiagnostics 교체 주석을 유지한다.

13. 재진입
- 전환 시작부터 OnTagUpdated 종료까지 동일 TagID 가드를 유지한다.
- Updated 콜백에서 같은 ID의 다음 호출은 false여야 한다.
-- TagAdd
-- TagSub
-- PerishableTagAdd
-- TagActivate
- 상태 변경 및 추가 이벤트가 없어야 한다.
- 다른 ID의 Add/Sub/PerishableTagAdd/Activate는 허용한다.
- 조회 API는 허용한다.
- finally에서 가드를 반드시 해제한다.
- 이벤트 예외 이후에도 후속 독립 API가 정상 작동해야 한다.

14. Tick 연동
- 전환된 Instance는 IsPerishable == false이므로 이후 Tick에서 감소하지 않아야 한다.
- Tick 스냅샷에 포함되었더라도 실제 처리 직전 IsPerishable 재검사에서 제외되어야 한다.
- Tick 구조나 isTicking 정책을 변경하지 않는다.
- Tick 도중 다른 태그의 콜백으로 해당 Instance가 영구 전환되면 그 Tick에서 더 이상 감소시키거나 만료시키지 않는다.

15. 전환 조건과 RemainingTime
- 기존 Instance가 Dictionary에 존재하고 IsPerishable == true이면 RemainingTime 크기와 관계없이 전환을 허용한다.
- RemainingTime이 0 또는 TimeEpsilon 근처여도 아직 만료 제거가 완료되지 않았다면 영구 전환할 수 있다.
- 전환 후 RemainingTime은 0f이다.
- 전환 후 IsPerishable은 false이므로 0f는 만료 상태가 아니라 영구 태그 상태를 의미한다.

16. PerishableTagAdd 임시 정책 유지
- 기존 영구 태그 + PerishableTagAdd: false
- 기존 소멸 태그 + PerishableTagAdd: false
- 상태, 시간, 스택, Trait, 이벤트 변화 없음
- 같은 ID 다른 Definition: Error + false
- 이후 단계 인계:
-- 단계 8: Refresh
-- 단계 9: MaxDuration
-- 단계 11: 소멸 StackCount

17. 기존 테스트 정책 변경
- 기존 영구 태그에 대한 동일 Definition 중복 TagAdd false 테스트는 유지한다.
- 기존 소멸 태그에 대한 TagAdd false 기대가 있다면 다음으로 의도적으로 수정한다.
-- 결과 true
-- 동일 Instance 영구 전환
-- ChangedToPermanent 이벤트 한 번
- 테스트를 삭제하거나 무조건 완화하지 않는다.
- 변경 이유를 코드 주석 또는 완료 보고에 기록한다.
- 단계 6의 PerishableTagAdd 중복 false 테스트는 유지한다.

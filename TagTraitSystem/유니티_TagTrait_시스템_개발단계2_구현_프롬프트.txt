1. AI 역할
- Unity 6 환경의 태그 보유 컴포넌트와 조회 API를 작성하는 Unity C# 개발자로 작업한다.
- 현재 단계에서는 TagContainer의 기본 컬렉션 구조와 조회 기능만 구현한다.
- 공개 태그 추가, 제거, Trait 실행, 이벤트, 시간 처리 및 중첩 정책은 구현하지 않는다.
- Unity Editor 내부 컴파일과 Test Runner 실행은 수행했다고 간주하지 않는다.
- 실제 컴파일과 테스트 실행은 사용자가 Unity에서 수행한다.

5. AI 작업 완료 기준
- 지정된 Runtime 및 EditMode 테스트 파일이 생성되어 있다.
- TagContainer의 내부 컬렉션과 공개 조회 API가 요구사항과 일치한다.
- TryAddInstance가 internal로 구현되어 있다.
- TagScan이 얕은 스냅샷을 반환한다.
- AND/OR 입력 검증이 전체 배열 선검증 방식으로 구현되어 있다.
- 동일 ID의 다른 Definition 조회는 ID 기준으로 동작한다.
- 동일 ID의 다른 Definition 삽입은 오류와 false로 차단한다.
- 실제 Unity 컴파일과 테스트는 미실행 상태로 명확히 보고한다.

9. 내부 컬렉션

9.1. 필드
- 다음 형태의 readonly 필드를 사용한다.
-- (private readonly Dictionary<TagID, TagInstance> tagInstances = new Dictionary<TagID, TagInstance>();)
- 필드 선언 시 초기화한다.
- Unity 직렬화 필드로 만들지 않는다.
- 외부에 Dictionary 자체를 공개하지 않는다.
- public 또는 internal 프로퍼티로 Dictionary를 노출하지 않는다.
- Count 프로퍼티를 추가하지 않는다.

9.2. 컬렉션 규칙
- TagID.None은 Key로 사용할 수 없다.
- 하나의 TagID는 하나의 TagInstance만 가진다.
- Dictionary에 Null TagInstance를 넣지 않는다.
- 현재 단계에서 제거 메서드를 구현하지 않는다.

10. 내부 삽입 메서드

10.1. 시그니처
- 다음 시그니처를 정확히 사용한다.
-- (internal bool TryAddInstance(TagInstance instance))

10.2. 책임
- 이미 생성된 TagInstance를 내부 Dictionary에 안전하게 등록한다.
- 단계 3의 공개 TagAdd가 이 메서드를 재사용할 수 있도록 작성한다.
- 테스트 전용 public API로 만들지 않는다.
- Trait, 이벤트, StackPolicy 및 시간 처리를 수행하지 않는다.

10.3. 처리 순서
- instance가 Null이면 false를 반환한다.
- instance.Definition이 Null이면 false를 반환한다.
-- 정상적인 TagInstance 생성자로는 발생하지 않지만 방어적으로 처리한다.
- instance.Definition.TagID가 None이면 false를 반환한다.
- 같은 TagID의 기존 Instance가 있는지 확인한다.
- 기존 Instance가 없으면 Dictionary에 추가하고 true를 반환한다.
- 기존 Instance가 있으면 기존 상태를 유지하고 false를 반환한다.

10.4. 동일 ID 처리
- 기존 Instance의 Definition과 요청 Instance의 Definition이 같은 참조이면 일반 중복 삽입이다.
-- false 반환
-- 로그 없음
-- 기존 Instance 유지
- 같은 TagID이지만 서로 다른 TagDefinition 참조이면 데이터 오류이다.
-- Debug.LogError 호출
-- false 반환
-- 기존 Instance 유지
-- 새 Instance를 Dictionary에 넣지 않음
- 오류 로그 바로 위 또는 인접 위치에 다음 의미의 한글 주석을 작성한다.
-- 개발 단계 12에서 TagDiagnostics로 교체한다.
- 로그 문구는 영어로 작성한다.

11. TagScan

11.1. 시그니처
- 다음 시그니처를 정확히 사용한다.
-- (public IReadOnlyCollection<TagInstance> TagScan())

11.2. 동작
- 호출 시점의 TagInstance 구성에 대한 얕은 스냅샷을 반환한다.
- 태그가 없으면 Null이 아닌 빈 컬렉션을 반환한다.
- 빈 결과에는 (Array.Empty<TagInstance>())를 사용할 수 있다.
- 태그가 있으면 새로운 TagInstance 배열을 만들고 Dictionary Values를 복사한다.
- LINQ의 ToArray를 사용하지 않고 Values.CopyTo를 우선한다.
- 반환 이후 TagContainer에 태그가 추가되어도 기존 반환 컬렉션 구성은 변경되지 않는다.
- TagInstance 객체 자체는 복제하지 않는다.
- 반환 순서를 보장하지 않는다.
- 외부에서 내부 Dictionary를 수정할 수 없어야 한다.

11.3. 문서화
- 영어 XML 문서 주석에 다음 의미를 포함한다.
-- 호출 시점의 얕은 스냅샷
-- 반환 순서 미보장
-- TagInstance 참조 자체는 복제하지 않음

12. TagCheck

12.1. TagDefinition overload
- 다음 시그니처를 정확히 사용한다.
-- (public bool TagCheck(TagDefinition definition))
- definition이 Null이면 false를 반환한다.
- definition.TagID가 None이면 false를 반환한다.
- Dictionary에 같은 TagID가 있으면 true를 반환한다.
- 전달된 Definition과 저장된 Definition 참조가 달라도 ID가 같으면 true를 반환한다.
- 로그를 남기지 않는다.
- 상태를 변경하지 않는다.

12.2. TagID overload
- 다음 시그니처를 정확히 사용한다.
-- (public bool TagCheck(TagID tagID))
- tagID가 None이면 false를 반환한다.
- Dictionary에 Key가 있으면 true를 반환한다.
- 없으면 false를 반환한다.
- 로그를 남기지 않는다.
- 상태를 변경하지 않는다.

13. TagANDCheck

13.1. 시그니처
- 다음 시그니처를 정확히 사용한다.
-- (public bool TagANDCheck(params TagDefinition[] definitions))

13.2. 입력 검증
- definitions 배열 자체가 Null이면 false를 반환한다.
- 빈 배열이면 false를 반환한다.
- 배열 내부에 Null Definition이 하나라도 있으면 Debug.LogError 후 false를 반환한다.
- 배열 내부에 TagID.None Definition이 하나라도 있으면 Debug.LogError 후 false를 반환한다.
- 전체 배열을 먼저 검증한 뒤 논리 검사를 시작한다.
- 유효한 보유 태그가 앞에 있더라도 뒤에 잘못된 입력이 있으면 false를 반환한다.

13.3. 논리 처리
- 단일 Definition이면 일반 TagCheck와 같은 결과를 반환한다.
- 모든 TagID가 Dictionary에 존재해야 true이다.
- 하나라도 없으면 false이다.
- 같은 Definition 또는 같은 ID가 중복되어도 결과에 영향을 주지 않는다.
- 중복 제거를 위한 HashSet을 만들지 않는다.
- 입력 전체 검증 후에는 하나라도 없으면 즉시 false를 반환할 수 있다.

13.4. 로그
- Null 배열은 로그 없이 false를 반환한다.
- 빈 배열은 로그 없이 false를 반환한다.
- 배열 내부 Null은 Debug.LogError 후 false를 반환한다.
- 배열 내부 None ID는 Debug.LogError 후 false를 반환한다.
- 로그 문구는 영어로 작성한다.
- 직접 Debug.LogError를 사용하는 위치에는 단계 12에서 TagDiagnostics로 교체한다는 한글 주석을 작성한다.

14. TagORCheck

14.1. 시그니처
- 다음 시그니처를 정확히 사용한다.
-- (public bool TagORCheck(params TagDefinition[] definitions))

14.2. 입력 검증
- definitions 배열 자체가 Null이면 false를 반환한다.
- 빈 배열이면 false를 반환한다.
- 배열 내부에 Null Definition이 하나라도 있으면 Debug.LogError 후 false를 반환한다.
- 배열 내부에 TagID.None Definition이 하나라도 있으면 Debug.LogError 후 false를 반환한다.
- 전체 배열을 먼저 검증한 뒤 논리 검사를 시작한다.
- 앞부분에 보유 태그가 있어도 뒤에 잘못된 입력이 있으면 true로 조기 반환하지 않는다.

14.3. 논리 처리
- 단일 Definition이면 일반 TagCheck와 같은 결과를 반환한다.
- 하나 이상의 TagID가 Dictionary에 존재하면 true이다.
- 모두 없으면 false이다.
- 같은 Definition 또는 같은 ID가 중복되어도 결과에 영향을 주지 않는다.
- 중복 제거를 위한 HashSet을 만들지 않는다.
- 입력 전체 검증 후에는 하나를 발견하면 즉시 true를 반환할 수 있다.

14.4. 로그
- Null 배열은 로그 없이 false를 반환한다.
- 빈 배열은 로그 없이 false를 반환한다.
- 배열 내부 Null은 Debug.LogError 후 false를 반환한다.
- 배열 내부 None ID는 Debug.LogError 후 false를 반환한다.
- 로그 문구는 영어로 작성한다.
- 직접 Debug.LogError를 사용하는 위치에는 단계 12에서 TagDiagnostics로 교체한다는 한글 주석을 작성한다.

15. TryGetTag

15.1. 시그니처
- 다음 시그니처를 정확히 사용한다.
-- (public bool TryGetTag(TagDefinition definition, out TagInstance instance))

15.2. 동작
- 메서드 시작 시 instance를 Null로 설정한다.
- definition이 Null이면 false를 반환한다.
- definition.TagID가 None이면 false를 반환한다.
- Dictionary에 같은 TagID가 있으면 저장된 실제 TagInstance를 instance에 할당하고 true를 반환한다.
- 전달된 Definition과 저장된 Definition 참조가 달라도 ID가 같으면 조회에 성공한다.
- 존재하지 않으면 instance를 Null로 유지하고 false를 반환한다.
- 로그를 남기지 않는다.
- 상태를 변경하지 않는다.

15.3. 범위 제한
- 다음 공개 overload를 추가하지 않는다.
-- (TryGetTag(TagID tagID, out TagInstance instance))
- 필요하면 Dictionary.TryGetValue를 직접 사용하는 private 헬퍼는 만들 수 있으나 공개 API를 늘리지 않는다.

16.2. 테스트 기본 구조
- 테스트마다 GameObject를 생성한다.

- TagDefinition은 ScriptableObject.CreateInstance<TagDefinition>()로 생성한다.

- 생성한 GameObject와 ScriptableObject는 필드 또는 컬렉션에 보관한다.
- [TearDown]에서 Object.DestroyImmediate로 정리한다.
- 프로젝트 에셋 파일을 생성하지 않는다.

16.3. 테스트 helper

- 공용 Test Factory 클래스는 아직 만들지 않는다.
- Runtime에 테스트용 setter 또는 public API를 추가하지 않는다.

16.4. 로그 검증
- 동일 ID의 다른 Definition 삽입 오류와 AND/OR 잘못된 요소 로그는 UnityEngine.TestTools.LogAssert를 사용해 검증한다.
- 로그 메시지 전체를 지나치게 세밀하게 고정하지 않는다.
- LogType.Error와 핵심 의미를 검증한다.
- 단계 12에서 로그 문자열이 변경될 수 있으므로 구현 세부 문자열에 과도하게 결합하지 않는다.

- private 필드는 별도 접두어 없이 camelCase를 사용한다.
- 중괄호는 BSD 스타일을 사용한다.
- var를 사용하지 않는다.
- XML 문서 주석은 영어로 작성한다.
- 일반 코드 주석은 한글로 작성한다.
- 로그 문구는 영어로 작성한다.
- 테스트 메서드 이름은 영어로 작성한다.
- 기능 구간이 명확한 경우에만 #region을 사용한다.
- Nullable Reference Type을 도입하지 않는다.
- 불필요한 LINQ 사용을 피한다.

- 차이점, 영향 범위 및 권장 수정안을 보고한다.
- 단계 0 환경 테스트와 단계 1 테스트를 삭제하거나 변경하지 않는다.

26. 금지 사항
- 단계 3 이후 기능을 미리 구현하지 않는다.
- TagContainer에 Awake, Start, Update를 추가하지 않는다.
- Dictionary를 public, protected 또는 internal로 직접 노출하지 않는다.
- TryAddInstance를 public으로 만들지 않는다.
- 테스트 전용 이름의 Runtime API를 추가하지 않는다.
- TagScan에서 내부 Values 실시간 뷰를 그대로 반환하지 않는다.
- TagScan에서 TagInstance를 깊은 복사하지 않는다.
- AND/OR에서 입력 전체 검증 전에 논리 결과를 조기 반환하지 않는다.
- 동일 ID의 다른 Definition 삽입을 정상 중복으로 무시하지 않는다.
- 조회 API에서 Definition 참조 동일성을 요구하지 않는다.
- 기존 테스트를 삭제하거나 기대값을 완화하지 않는다.
- 실제 테스트를 실행하지 않고 통과했다고 보고하지 않는다.
- Unity 컴파일을 확인하지 않고 오류가 없다고 단정하지 않는다.

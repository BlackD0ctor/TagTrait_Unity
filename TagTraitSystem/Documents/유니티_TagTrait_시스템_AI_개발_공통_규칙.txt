유니티 TagTrait 시스템 AI 개발 공통 규칙

0. 문서 개요
- 본 문서는 유니티 TagTrait 시스템의 단계별 AI 구현 프롬프트와 검수 프롬프트에 공통으로 적용할 확정 규칙을 정리한 문서이다.
- 원본 개발 설계서는 전체 시스템의 기능, 개발 순서, 완료 조건을 정의하는 마스터 설계서로 유지한다.
- 본 공통 규칙 문서는 원본 설계서에서 AI 작업 방식, 프로젝트 환경, 구현 안전성, 데이터 유효성 및 테스트 책임을 구체화한다.
- 단계별 프롬프트는 본 문서와 원본 개발 설계서를 함께 참조하는 혼합형으로 작성한다.
- 각 단계에서만 필요한 세부 파일 목록, 정확한 내부 메서드명, 테스트 목록 및 구현 제외 범위는 해당 단계 프롬프트에서 확정한다.
- 본 문서와 원본 개발 설계서가 충돌하는 경우 본 문서에 명시된 변경 및 확정 사항을 우선한다.

0.1. 기준 문서 배치
- 시스템 루트 경로는 (Assets/TagTraitSystem)이다.
- 원본 개발 설계서와 본 공통 규칙 문서는 시스템 루트 경로 내부에 배치한다.
- 각 단계별 구현 프롬프트는 개발 순서에 따라 하나씩 전달한다.
- 각 단계별 검수 프롬프트는 해당 구현 프롬프트와 분리하여 작성한다.

0.2. 기준 충돌 우선순위
- 구현 중 기준이 충돌할 경우 다음 순서를 따른다.
-- 1순위: 현재 단계 프롬프트
-- 2순위: 본 공통 규칙 문서
-- 3순위: 마스터 개발 설계서
-- 4순위: 이전 단계 완료 기록
-- 5순위: 현재 코드와 테스트
-- 6순위: 일반적인 Unity 또는 C# 관례
- 충돌이 발견되면 AI는 임의로 기준을 선택하여 구현하지 않는다.
- 충돌 내용, 영향 범위, 변경이 필요한 파일과 공개 API를 먼저 보고한다.
- 기존 테스트와 최신 설계가 충돌하는 경우 테스트 기대값을 임의로 변경하거나 완화하지 않는다.

1. 단계별 프롬프트 운용 규칙

1.1. 프롬프트 구성 방식
- 프롬프트는 혼합형으로 작성한다.
- 공통 규칙과 원본 개발 설계서는 프로젝트에 상시 배치한다.
- 단계별 프롬프트에는 현재 단계에 필요한 확정 동작, 파일, API, 테스트, 제외 범위를 다시 명시한다.
- AI는 다른 단계의 기능을 참고할 수 있으나 현재 단계에서 조기 구현하지 않는다.

1.2. 단계 분할 원칙
- 기본적으로 하나의 개발 단계에는 하나의 구현 프롬프트를 사용한다.
- 작업 규모가 크거나 서로 다른 책임이 혼재하는 경우 프롬프트를 분할할 수 있다.
- 분할 번호는 (N-n) 형식을 사용한다.
-- 예시: (5-1), (5-2)
- 분할은 결합도와 응집도를 기준으로 한다.
- 하나의 프롬프트는 가능한 한 하나의 명확한 책임과 검증 범위를 가져야 한다.
- 분할된 모든 프롬프트가 완료되어야 해당 원래 단계의 AI 작업이 완료된 것으로 본다.

1.3. 구현 프롬프트와 검수 프롬프트
- 구현 프롬프트와 검수 프롬프트는 분리한다.
- 구현 프롬프트는 코드 및 테스트 코드 작성에 사용한다.
- 검수 프롬프트는 새 기능 구현보다 설계 일치성, 범위 초과, 테스트 누락 및 공개 API 변경 여부를 확인하는 데 사용한다.
- 검수 과정에서 수정이 필요한 경우 수정 대상과 이유를 먼저 구분하여 보고한다.

1.4. 단계 상태 전달
- 각 단계 프롬프트에는 이전 단계 완료 기록을 포함한다.
- 이전 단계 완료 기록에는 다음 내용을 포함한다.
-- 완료한 개발 단계
-- 생성 및 수정한 파일
-- 추가된 공개 API
-- 작성한 테스트
-- 사용자가 확인한 컴파일 결과
-- 사용자가 확인한 테스트 결과
-- 실패 또는 보류 항목
-- 승인된 설계 변경
-- 다음 단계 진행 가능 여부
- 사용자가 Unity에서 확인하지 않은 컴파일 및 테스트 결과는 완료 기록에 성공으로 기재하지 않는다.

2. AI 작업 범위와 완료 기준

2.1. AI 작업 범위
- AI는 현재 단계의 Runtime 코드, Editor 코드 또는 테스트 코드를 프롬프트 범위 내에서 작성한다.
- AI는 실제 프로젝트 파일을 생성하거나 수정할 수 있다.
- AI는 Unity Test Framework용 테스트 코드를 작성한다.
- AI는 코드와 테스트 코드에 대한 정적 검토를 수행한다.
- AI는 Unity Editor 내부의 컴파일, Test Runner 실행, ScriptableObject 에셋 생성 및 Scene 생성을 완료했다고 주장하지 않는다.
- AI는 Unity에서 사용자가 추가하거나 생성해야 할 요소를 완료 보고에 구체적으로 기록한다.
- AI는 Git commit, Git tag 또는 릴리스 태그를 임의로 생성하지 않는다.
- Git 관련 작업은 해당 단계 프롬프트에서 명시적으로 허용된 경우에만 수행한다.

2.2. AI 작업 완료
- 다음 조건을 만족하면 해당 단계의 AI 작업이 완료된 것으로 본다.
-- 현재 단계 코드 작성 완료
-- 현재 단계 테스트 코드 작성 완료
-- 변경 범위 정적 검토 완료
-- 사용자가 수행해야 할 Unity 작업 목록 보고 완료
-- 컴파일 및 테스트가 미실행 상태임을 명확히 보고

2.3. 개발 단계 최종 완료
- 다음 조건을 만족하면 해당 개발 단계가 최종 완료된 것으로 본다.
-- 사용자가 Unity에서 프로젝트 컴파일 성공을 확인
-- 사용자가 현재 단계 테스트를 실행하고 통과를 확인
-- 사용자가 기존 전체 회귀 테스트를 실행하고 통과를 확인
-- 발견된 오류 또는 설계 충돌이 해결됨
- AI가 코드를 작성했다는 사실만으로 개발 단계가 최종 완료된 것으로 판정하지 않는다.

2.4. 결과 보고 표현
- 실제 실행하지 않은 테스트를 통과했다고 보고하지 않는다.
- Unity 컴파일을 확인하지 않았다면 컴파일 성공이라고 보고하지 않는다.
- 미실행 상태는 다음과 같이 구분하여 보고한다.
-- Test code written, not executed.
-- Unity compilation not verified.
-- Manual Unity verification required.

3. 개발 환경

3.1. Unity 환경
- Unity 버전은 (Unity 6.0 LTS, 6000.0.68f1)이다.
- API Compatibility Level은 (.NET Standard 2.1)이다.
- Unity Test Framework 버전은 (1.6.0)이다.
- 최소 지원 플랫폼은 PC이다.
- 1순위 개발 및 검증 환경은 Windows PC이다.
- 최종 확장 목표에는 모바일 환경을 포함한다.
- 빌드 백엔드로 IL2CPP 사용을 고려한다.
- Reflection, 동적 코드 생성 또는 IL2CPP에서 문제가 될 수 있는 기능을 도입할 경우 먼저 영향 범위를 보고한다.

3.2. 경로
- 시스템 루트 경로: (Assets/TagTraitSystem)
- Runtime 코드: (Assets/TagTraitSystem/Runtime)
- Editor 코드: (Assets/TagTraitSystem/Editor)
- EditMode 테스트: (Assets/TagTraitSystem/Tests/EditMode)
- PlayMode 테스트: (Assets/TagTraitSystem/Tests/PlayMode)
- Sample 코드 및 자료: (Assets/TagTraitSystem/Samples)

3.3. 네임스페이스
- 기본 네임스페이스는 다음과 같다.
-- (TagTraitSystem.Runtime)
-- (TagTraitSystem.Editor)
-- (TagTraitSystem.Tests)
-- (TagTraitSystem.Samples)
- 클래스 책임 분리를 위해 한 단계의 하위 네임스페이스를 사용할 수 있다.
-- 예시: (TagTraitSystem.Runtime.Core)
-- 예시: (TagTraitSystem.Runtime.Definitions)
-- 예시: (TagTraitSystem.Runtime.Traits)
-- 예시: (TagTraitSystem.Runtime.Components)
- 필요 이상으로 하위 네임스페이스를 중첩하지 않는다.
- 단순한 (Runtime), (Editor), (Tests), (Samples)를 최상위 네임스페이스로 사용하지 않는다.

3.4. Assembly Definition
- 개발 단계 0에서 Runtime Assembly Definition을 먼저 구성한다.
- 개발 단계 0에서 EditMode Tests Assembly Definition을 구성하고 Runtime Assembly를 참조한다.
- Editor Assembly Definition은 Editor 코드가 처음 추가되는 단계에 구성한다.
- PlayMode Tests Assembly Definition은 PlayMode 테스트가 처음 추가되는 단계에 구성한다.
- Editor 및 Test Assembly가 Runtime Assembly를 참조하도록 한다.
- Runtime Assembly가 Editor 또는 Test Assembly를 참조하지 않는다.
- Assembly Definition의 정확한 이름과 참조 목록은 단계 0 프롬프트에서 확정한다.

4. 코딩 규칙

4.1. 필드와 프로퍼티
- private 필드는 별도 접두어 없이 camelCase를 사용한다.
-- 예시: (private TagID tagID;)
- Inspector에서 자주 수정하거나 에셋 데이터로 저장해야 하는 필드 위주로 직렬화한다.
- 직렬화 필드는 기본적으로 다음 형식을 사용한다.
-- ([SerializeField] private TagID tagID;)
- 외부 공개는 읽기 전용 프로퍼티를 우선한다.
-- 예시: (public TagID TagID => tagID;)
- Runtime 상태를 외부가 임의로 변경할 수 있는 public setter를 제공하지 않는다.

4.2. 코드 스타일
- 중괄호는 BSD 스타일을 사용한다.
- (var)를 사용하지 않는다.
- 기능 단위로 구분이 필요한 경우에만 (#region)을 사용한다.
- region 이름은 기능을 설명하도록 작성한다.
-- 예시: (#region Tag Query)
- 지나치게 작은 메서드 또는 필드 단위로 region을 남용하지 않는다.
- Nullable Reference Type은 필요 시 도입할 수 있으나, 도입 전에 적용 범위와 기존 코드 영향을 보고한다.

4.3. 언어 규칙
- XML 문서 주석은 영어로 작성한다.
- 일반 코드 주석은 한글로 작성한다.
- Runtime 및 Editor 로그 문구는 영어로 작성한다.
- 테스트 메서드 이름은 영어로 작성한다.
- 테스트 이름은 가능한 한 ({대상}_{조건}_{기대결과}) 형식을 따른다.

4.4. 생성자와 접근 제한자
- 일반 Runtime 클래스의 생성자는 특별한 제약이 없으면 public으로 작성한다.
- 상태 변경을 제한해야 하는 생성자 또는 메서드는 internal 또는 private으로 제한할 수 있다.
- 접근 제한을 변경해야 할 경우 캡슐화 이유와 테스트 영향 범위를 보고한다.
- Test Assembly에서 Runtime internal 멤버에 접근해야 할 경우 (InternalsVisibleTo)를 사용할 수 있다.

5. 테스트 공통 규칙

5.1. 테스트 작성
- 각 구현 단계에서 해당 기능의 테스트 코드를 함께 작성한다.
- 테스트는 정상 흐름, 경계값, 실패 흐름을 포함한다.
- 현재 단계에서 아직 구현하지 않는 기능을 테스트 성공 조건으로 요구하지 않는다.
- 기존 테스트를 삭제하거나 기대값을 완화하여 통과시키지 않는다.
- 테스트 실행은 사용자가 Unity Test Runner에서 수행한다.

5.2. Unity Object 생성과 정리
- 테스트용 ScriptableObject는 (ScriptableObject.CreateInstance<T>())로 생성할 수 있다.
- 테스트 중 생성한 GameObject와 ScriptableObject는 테스트 종료 시 정리한다.
- EditMode 테스트에서는 ([TearDown])에서 (Object.DestroyImmediate)를 사용할 수 있다.
- 테스트 중 메모리에 생성한 Unity Object와 프로젝트 에셋 파일을 구분한다.
- AI는 프로젝트 에셋 파일을 임의로 삭제하지 않는다.
- Scene 또는 에셋 파일 생성이 필요하면 사용자가 수행할 작업으로 보고한다.

5.3. 테스트용 TagID
- 초기 TagID에는 다음 테스트용 값을 포함한다.
-- (None = 0)
-- (TestAlpha = 1)
-- (TestBeta = 2)
-- (TestGamma = 3)
- 테스트용 값임을 코드 주석으로 표시한다.
- 테스트용 TagID는 초기 개발 및 검증을 위한 임시 값이다.
- 제거 또는 실제 프로젝트 ID로 교체하는 시점은 초기 버전 정리 단계에서 검토한다.

5.4. 단계 2 테스트 데이터 주입
- 개발 단계 2에서는 공개 (TagAdd)가 아직 구현되지 않는다.
- 조회 테스트를 위해 Runtime 내부의 핵심 삽입 메서드를 구현한다.
- 해당 메서드는 테스트만을 위한 public API로 만들지 않는다.
- 단계 3의 공개 (TagAdd)는 단계 2에서 만든 내부 삽입 메서드를 재사용한다.
- 정확한 내부 메서드명과 시그니처는 단계 2 프롬프트에서 확정한다.
- 내부 삽입 메서드는 Dictionary 캡슐화를 유지해야 한다.

6. Enum 및 기본값

6.1. TagID
- (None = 0)은 실제 태그로 사용할 수 없다.
- 초기 테스트용 ID는 다음과 같다.
-- (TestAlpha = 1)
-- (TestBeta = 2)
-- (TestGamma = 3)
- 서로 다른 enum 이름에 같은 숫자값을 지정하지 않는다.

6.2. TagCategory
- 값은 다음과 같이 고정한다.
-- (None = 0)
-- (Keyword = 1)
-- (Status = 2)
- 이후 카테고리는 실제 필요가 생겼을 때 추가한다.
- TagCategory는 태그 분류 및 키워드 태그 판정에 사용한다.
- 태그 동일성 판정에는 사용하지 않는다.

6.3. StackPolicy
- 값은 다음과 같이 고정한다.
-- (None = 0)
-- (MaxDuration = 1)
-- (Refresh = 2)
-- (StackCount = 3)
- (None)은 중첩 동작이 없는 기본 정책이다.
- (Extend)는 실제 구현 단계 전까지 추가하지 않는다.

6.4. TagChangeReason
- 값은 다음과 같이 고정한다.
-- (Added = 0)
-- (Removed = 1)
-- (DurationRefreshed = 2)
-- (DurationExtendedToMax = 3)
-- (StackIncreased = 4)
-- (StackDecreased = 5)
-- (ChangedToPermanent = 6)
-- (Expired = 7)

6.5. ExclusiveGroup
- 초기 값은 (None = 0)만 구현한다.
- ExclusiveGroup은 그룹 식별용이며 우선순위를 의미하지 않는다.
- 실제 상호 배타 처리 알고리즘은 보류한다.
- 우선순위는 별도의 (Priority) 정수 필드로 유지한다.

7. TagDefinition 공통 규칙

7.1. 보류 필드
- 다음 필드는 원본 설계서대로 초기부터 포함한다.
-- (ExclusiveGroup)
-- (Priority)
-- (IsSaveable)
- 초기 구현에서는 해당 필드의 실제 동작을 구현하지 않는다.
- 보류 필드를 사용 가능하다고 가정하여 Runtime 로직을 작성하지 않는다.

7.2. Trait 필드 도입
- 개발 단계 1에서 (TagTraitSO) 최소 타입과 TagDefinition의 Trait 필드를 추가한다.
- 개발 단계 1의 TagTraitSO는 TagDefinition이 컴파일될 수 있는 최소 형태로 작성한다.
- (ITagTrait), Trait 생명주기 및 TagActivate 동작은 개발 단계 5에서 완성한다.

7.3. 기본 유효성
- TagID가 None이면 유효하지 않다.
- 태그 이름이 빈 문자열이면 유효하지 않다.
- 태그 이름이 공백만으로 구성된 경우는 허용한다.
- 태그 이름의 앞뒤 공백은 허용한다.
- 중복 이름은 Editor Validator에서 Warning으로 보고한다.
- 중복 ID는 데이터 오류로 처리한다.

7.4. 최대 스택 설정
- 태그별 최대 스택 설정의 원본은 TagDefinition이다.
- 개발 단계 1에서 TagDefinition에 다음 직렬화 필드를 추가한다.
-- ([SerializeField] private int maxStackCount = 1;)
- 외부에는 읽기 전용 (MaxStackCount) 프로퍼티로 공개한다.
- (MaxStackCount)는 (StackPolicy.StackCount) 동작에서만 사용한다.
- 다른 StackPolicy에서도 직렬화된 값은 보존하지만 Runtime 정책에는 사용하지 않는다.
- 기본값은 (1)이다.
- (MaxStackCount == 1)은 유효한 설정이다.
-- 최초 한 개의 태그만 보유할 수 있다.
-- 중복 추가는 최대치 도달로 처리한다.
- (MaxStackCount < 1)은 잘못된 데이터이다.
- TagInstance에 최대 스택 수를 별도 Runtime 상태로 복제하거나 저장하지 않는다.
- 최대 스택 수는 TagDefinition에서 직접 조회한다.
- 같은 TagDefinition을 사용하는 오브젝트별 최대치 변경 또는 Runtime override는 본 시스템 범위에서 지원하지 않는다.
- 오브젝트별 최대치가 필요하면 사용자가 별도 오브젝트 또는 스크립트에서 확장한다.

8. TagID 유일성 및 중복 처리

8.1. 프로젝트 전역 유일성
- TagID는 프로젝트 전체에서 TagDefinition을 식별하는 고유 ID이다.
- 하나의 TagID에는 하나의 TagDefinition 에셋만 존재해야 한다.
- 서로 다른 TagDefinition 에셋이 같은 TagID를 사용하는 것은 데이터 오류이다.

8.2. Runtime Container 검증
- Runtime Container에서 같은 TagID의 기존 TagInstance를 발견하면 기존 Instance의 Definition과 요청 Definition을 비교한다.
- 같은 TagID이며 같은 TagDefinition 참조이면 정상적인 중복 추가, 시간 갱신 또는 스택 정책을 적용한다.
- 같은 TagID이지만 서로 다른 TagDefinition 참조이면 오류를 출력하고 false를 반환한다.
- Definition 불일치 시 기존 상태를 변경하지 않는다.
- Definition 불일치 시 Trait을 호출하지 않는다.
- Definition 불일치 시 태그 변경 이벤트를 발생시키지 않는다.
- 이 검증은 다음 변경 및 실행 API에 적용한다.
-- (TagAdd)
-- (TagSub)
-- (PerishableTagAdd)
-- (TagActivate)

8.3. 조회 API
- (TagCheck)와 (TryGetTag)는 TagID 기준으로 조회한다.
- 동일 ID의 다른 Definition이 전달되어도 조회 자체는 ID를 기준으로 수행할 수 있다.
- 조회 결과로 중복 Definition 사용이 정상 데이터라는 의미가 되지는 않는다.

8.4. Editor 검증
- 프로젝트 전체 TagDefinition 에셋의 중복 ID 검사는 Editor Validator가 수행한다.
- 같은 TagID를 가진 에셋이 둘 이상이면 Error로 보고한다.
- 같은 이름을 가진 에셋이 둘 이상이면 Warning으로 보고한다.
- OnValidate만으로 프로젝트 전체 에셋의 중복 ID를 확정하지 않는다.

9. 키워드 태그 규칙

9.1. 판정 기준
- 키워드 태그 여부는 (TagCategory.Keyword)만으로 판정한다.
- Trait의 Null 여부는 키워드 태그 판정에 사용하지 않는다.
- (IsKeywordTag)는 TagCategory를 기준으로 반환한다.

9.2. 정상 데이터 작성 규칙
- TagCategory가 Keyword인 TagDefinition에는 Trait을 연결하지 않는다.
- TagCategory가 Keyword가 아닌 실행형 태그에는 필요한 Trait을 연결한다.

9.3. Keyword에 Trait이 연결된 경우
- (Category == Keyword && Trait != null)은 유효하지 않은 TagDefinition이다.
- (TagAdd)는 오류를 출력하고 false를 반환한다.
- (PerishableTagAdd)는 오류를 출력하고 false를 반환한다.
- 잘못된 Keyword 태그는 새 TagInstance를 생성하지 않는다.
- 상태 변경, Trait 호출 및 태그 변경 이벤트를 수행하지 않는다.
- (TagActivate)는 Trait이 연결되어 있어도 실행하지 않고 false를 반환한다.
- 비정상 Keyword 태그가 이미 Container에 존재하는 경우 (TagSub)를 통한 제거는 허용한다.
- 비정상 Keyword 태그 제거 시 연결된 Trait의 (OnRemove)는 호출하지 않는다.

9.4. 비Keyword 태그의 Trait Null
- (Category != Keyword && Trait == null)은 실행 가능한 Trait이 없는 유효하지 않은 설정이다.
- (TagActivate)는 오류를 출력하고 false를 반환한다.
- Null Trait을 호출하지 않는다.
- 해당 불일치는 단계 12의 Editor Validator 검증 대상으로 포함할 수 있다.
- TagAdd 단계에서 비Keyword Trait Null을 차단할지 여부는 해당 단계 프롬프트에서 원본 설계 의도와 함께 최종 확인한다.

10. TagScan 규칙

10.1. 반환 형태
- 공개 API는 원본 설계서의 다음 형태를 유지한다.
-- (IReadOnlyCollection<TagInstance> TagScan())
- TagScan은 TagID 목록만이 아니라 현재 보유 중인 TagInstance 목록을 반환한다.
- 태그가 없으면 Null이 아닌 빈 컬렉션을 반환한다.

10.2. 얕은 스냅샷
- TagScan은 호출 시점에 Container가 보유한 TagInstance 구성의 얕은 스냅샷을 반환한다.
- 이후 TagContainer의 태그 추가 또는 제거는 이미 반환된 컬렉션 구성에 반영되지 않는다.
- 반환 컬렉션 내부의 TagInstance는 복제하지 않는다.
- TagInstance는 참조형이므로 RemainingTime, StackCount 등의 상태 변경은 기존 반환 컬렉션에서 보일 수 있다.
- 반환 순서는 보장하지 않는다.
- 외부 코드는 반환 컬렉션을 통해 TagContainer 내부 Dictionary를 수정할 수 없어야 한다.

10.3. 성능 확장
- 초기 구현에서는 안전한 스냅샷을 우선한다.
- 실제 프로파일링에서 할당 문제가 확인되기 전에는 실시간 뷰를 공개 계약으로 추가하지 않는다.
- 추후 필요하면 호출자가 List를 재사용하는 복사 API 등 별도 저할당 확장을 설계한다.

11. float 및 시간 비교 규칙

11.1. Duration 비교
- 입력을 정규화한 Duration 값은 정확 비교를 사용할 수 있다.
- 시간값은 소수점 첫째 자리까지 정규화한다.
- 중간값은 0에서 멀어지는 방향으로 반올림한다.

11.2. RemainingTime 비교
- Tick으로 변화하는 RemainingTime 비교에는 epsilon을 사용한다.
- 이벤트 발생 여부 등 실제 상태 변화 판정에도 epsilon을 사용한다.
- 정확한 epsilon 값 또는 비교 헬퍼는 해당 시간 기능 단계 프롬프트에서 확정한다.
- 부동소수점 오차만으로 OnTagUpdated가 발생하지 않도록 한다.

12. Tick 및 StackCount 소멸 규칙

12.1. 큰 Tick 처리
- 하나의 소멸 TagInstance가 여러 StackCount를 가진 경우 큰 deltaTime은 여러 스택에 연속 적용한다.
- 하나의 Tick에서 경과 시간이 충분하면 같은 TagInstance의 여러 스택을 소모할 수 있다.
- 초과 시간을 버리지 않는다.
- 큰 Tick 한 번과 동일 총시간의 작은 Tick 여러 번이 가능한 한 같은 최종 상태를 만들어야 한다.

12.2. 일부 스택이 남는 경우
- Tick 처리 후 StackCount가 1 이상 남으면 TagInstance를 유지한다.
- 초과 시간을 반영한 최종 RemainingTime을 저장한다.
- OnRemove를 호출하지 않는다.
- OnTagUpdated를 (StackDecreased) 이유로 한 번만 발생시킨다.

12.3. 모든 스택이 소진되는 경우
- 모든 스택이 소진되면 TagInstance를 완전히 제거한다.
- OnRemove를 한 번 호출한다.
- Dictionary에서 제거한다.
- OnTagRemoved를 (Expired) 이유로 한 번 호출한다.
- 중간 스택마다 OnTagUpdated, OnRemove 또는 OnTagRemoved를 호출하지 않는다.

12.4. 최대 스택 적용
- 최대 스택 수는 (TagDefinition.MaxStackCount)를 사용한다.
- 현재 StackCount가 MaxStackCount보다 작을 때만 한 스택을 추가한다.
- 현재 StackCount가 MaxStackCount 이상이면 추가 요청을 무시한다.
- 최대치 도달은 정상적인 게임 로직 결과로 처리한다.
-- false 반환
-- 상태 변화 없음
-- Trait 호출 없음
-- 태그 변경 이벤트 없음
-- 오류 로그 없음
- (MaxStackCount < 1)인 StackCount 정책 데이터는 잘못된 설정이다.
-- Runtime 변경 API는 오류를 보고하고 false를 반환한다.
-- 상태, Trait 및 이벤트를 변경하지 않는다.
- 개발 중 Definition 설정 변경 또는 비정상 데이터로 (StackCount > MaxStackCount)가 되어도 자동으로 StackCount를 축소하지 않는다.
- 최대치를 초과한 기존 상태는 그대로 유지하고 추가 요청만 차단한다.
- 초과 상태와 잘못된 최대치는 개발 단계 12의 Diagnostics 및 Editor Validator에서 보고한다.
- Definition 설정 변경만으로 자동 StackDecreased 이벤트를 발생시키지 않는다.

13. 이벤트 집계 규칙

13.1. 공통 집계 원칙
- 하나의 공개 API 호출 또는 하나의 Tick 처리에서 같은 TagID에 여러 내부 상태 변경이 발생해도 외부 변경 이벤트는 최종 결과를 기준으로 최대 한 번 발생시킨다.
- TagInstance가 최종적으로 남아 있으면 OnTagUpdated를 한 번 발생시킨다.
- TagInstance가 최종적으로 제거되면 중간 OnTagUpdated 없이 OnTagRemoved를 한 번 발생시킨다.
- Trait 생명주기 호출도 중간 스택 단위로 반복하지 않는다.

13.2. Previous 값
- TagChangeEventData의 Previous 값은 해당 공개 API 또는 Tick 호출 직전 상태를 기준으로 한다.
- 하나의 호출 내부에서 발생한 중간 상태는 외부 이벤트로 노출하지 않는다.
- 스택 감소량은 PreviousStackCount와 최종 Instance.StackCount의 차이로 판단할 수 있어야 한다.

14. 재진입 규칙

14.1. 도입 시점
- 최소 재진입 가드는 개발 단계 4에서 추가한다.
- 내부 (HashSet<TagID>)을 사용하여 현재 처리 중인 TagID를 추적한다.
- 단계 12에서는 재진입 로그, 검증 및 테스트를 강화한다.

14.2. 허용 및 차단
- 같은 TagContainer에서 현재 처리 중인 동일 TagID에 대한 변경 재진입은 차단한다.
- 동일 TagID 재진입 요청은 false를 반환한다.
- 서로 다른 TagID에 대한 변경은 허용한다.
- 조회 API 호출은 상태를 변경하지 않으므로 별도 차단 대상이 아니다.
- 재진입 차단 중 상태가 절반만 변경된 채 남지 않도록 처리 순서를 유지한다.

15. 로그 및 Diagnostics 규칙

15.1. 단계 12 이전
- 단계 12 이전에는 디버깅 및 데이터 오류 추적에 필요한 최소한의 (Debug.LogError)를 직접 사용할 수 있다.
- 단계 12에서 TagDiagnostics로 교체할 대상임을 코드 주석으로 표시한다.
- 일반 성공 로그를 단계 12 이전에 과도하게 추가하지 않는다.

15.2. 단계 12
- 단계 12에서 공통 로그 출력을 TagDiagnostics로 정리한다.
- 직접 사용한 교체 대상 Debug.LogError를 TagDiagnostics 호출로 전환한다.
- 로그 문구는 영어로 작성한다.
- 데이터 손상 가능성이 있는 오류는 Diagnostics 활성화 여부와 관계없이 출력할 수 있다.
- StackPolicy가 StackCount인데 MaxStackCount가 1 미만이면 잘못된 설정으로 보고한다.
- MaxStackCount가 1인 설정은 유효하며 필수 Warning으로 취급하지 않는다.
- Runtime TagInstance의 StackCount가 Definition.MaxStackCount를 초과하면 잘못된 상태로 보고한다.
- 최대치 초과 상태를 Diagnostics가 자동으로 보정하거나 축소하지 않는다.

16. Unity 에셋 및 Sample 처리

16.1. 에셋 생성
- AI는 Unity Editor에서 ScriptableObject 에셋, Scene 또는 Inspector 설정을 실제로 생성했다고 간주하지 않는다.
- 코드상 생성 도구가 별도로 요구되지 않는 한 Unity YAML 에셋 또는 Scene 파일을 임의로 직접 작성하지 않는다.
- 사용자가 생성해야 할 에셋, 컴포넌트, Inspector 값 및 Scene 구성을 완료 보고에 목록으로 작성한다.

16.2. Sample
- Sample Scene과 Sample용 ScriptableObject 에셋은 사용자가 Unity에서 생성한다.
- AI는 필요한 Sample 스크립트와 생성 절차를 작성할 수 있다.
- 정확한 Sample 구성은 개발 단계 13 프롬프트에서 확정한다.

- 반환값
- 실패 시 상태
- 예외 시 상태
- 로그 처리

17.7. 비범위
- 현재 단계에서 구현하지 않을 기능
- 조기 생성하지 않을 타입
- 호출하지 않을 Trait 또는 이벤트
- 추가하지 않을 추상화
- 사용할 수 있다고 가정하면 안 되는 보류 기능

17.8. 테스트
- 작성할 테스트 이름
- 준비 상태
- 호출 API

- 기대 이벤트와 금지 이벤트

18. AI 금지 사항

18.1. 구현 금지
- 전체 시스템을 한 번에 구현하지 않는다.
- 현재 단계 밖의 기능을 조기 구현하지 않는다.
- 아직 검증되지 않은 추상 계층을 과도하게 추가하지 않는다.
- 보류 기능을 사용할 수 있다고 가정하지 않는다.
- Runtime 상태를 TagDefinition 또는 Trait ScriptableObject 에셋에 저장하지 않는다.
- 외부 코드가 TagContainer의 내부 Dictionary를 직접 수정할 수 있게 하지 않는다.

18.2. 테스트 금지
- 기존 테스트를 삭제하지 않는다.
- 설계와 충돌한다는 이유만으로 기대값을 임의로 완화하지 않는다.
- 실행하지 않은 테스트를 통과했다고 보고하지 않는다.
- Unity 컴파일을 확인하지 않고 오류가 없다고 단정하지 않는다.

18.3. 변경 금지
- 공개 API를 임의로 변경하지 않는다.
- 관련 없는 리팩터링을 수행하지 않는다.
- 다른 시스템 또는 프로젝트 전역 설정을 현재 단계 요구 없이 변경하지 않는다.
- 사용자의 승인 없이 NRT, 외부 라이브러리 또는 새로운 직렬화 방식을 도입하지 않는다.

19. 구현 중단 및 보고 조건
- 현재 단계 프롬프트와 공통 규칙 또는 마스터 설계서가 충돌하는 경우
- 공개 API 파괴 변경이 필요한 경우
- 현재 단계 밖의 파일 수정이 필수인 경우
- Unity 6000.0.68f1 또는 .NET Standard 2.1 환경에서 요구 기능을 사용할 수 없는 경우
- IL2CPP 호환성 문제가 예상되는 경우
- 테스트 환경 또는 Assembly 참조 구성이 불가능한 경우
- 설계서에 없는 정책 결정이 필요한 경우
- 기존 직렬화 데이터의 마이그레이션이 필요한 경우
- 상태 일관성을 보장할 수 없는 재진입 문제가 발생한 경우
- AI는 위 상황에서 임의 해결하지 않고 문제, 선택지, 권장안 및 영향 범위를 보고한다.

20. 공통 규칙 확정 상태
- 본 문서의 규칙은 단계별 AI 구현 프롬프트를 작성하기 위한 공통 기준으로 확정한다.
- 단계별로만 필요한 구체적인 내부 메서드명, CreateAssetMenu 값, epsilon 상수, 테스트 파일명, Sample 구성 등은 해당 단계 프롬프트에서 별도로 확정한다.
- 단계별 세부 결정이 본 공통 규칙의 의미를 변경해야 하는 경우 먼저 공통 규칙 개정 필요 여부를 보고한다.

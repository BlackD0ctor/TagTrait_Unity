1. AI 역할
- Unity 6 환경에서 영구 태그의 기본 추가 및 제거 기능을 작성한다.
- 현재 단계에서는 TagContainer의 공개 TagAdd와 TagSub만 구현한다.
- StackCount 중첩, Trait 생명주기, 이벤트, 소멸 태그 및 시간 처리는 구현하지 않는다.
- Unity 컴파일과 Test Runner 실행은 사용자가 수행한다.

2. 개발 환경
- Unity 6.0 LTS (6000.0.68f1)
- .NET Standard 2.1
- Unity Test Framework 1.6.0
- Runtime Assembly: TagTraitSystem.Runtime
- EditMode Test Assembly: TagTraitSystem.Tests.EditMode
- IL2CPP 호환성을 고려한다.

3. 이전 단계 전제
- 단계 1의 데이터 구조와 단계 2의 TagContainer 조회 기능이 Unity 검증을 통과한 것으로 전제한다.
- TagContainer에는 다음 내부 메서드가 존재해야 한다.
-- internal bool TryAddInstance(TagInstance instance)
- 다음 조회 API가 존재해야 한다.
-- TagScan
-- TagCheck overload
-- TagANDCheck
-- TagORCheck
-- TryGetTag
- 전제가 충족되지 않으면 구현을 시작하지 말고 누락 내용을 보고한다.

4. 현재 단계 목표
- 영구 태그를 추가하는 TagAdd 구현
- 보유 태그를 완전히 제거하는 TagSub 구현
- TagDefinition 입력 유효성 검사
- 잘못된 Keyword-Trait 조합과 비Keyword-Trait Null 조합 차단
- 같은 ID의 다른 TagDefinition 요청 차단
- TryAddInstance 재사용
- 영구 태그 CRUD EditMode 테스트 작성
- TagScan 제거 이후 스냅샷 테스트 작성

6. 공개 API

6.1. TagAdd
- 정확한 시그니처:
-- public bool TagAdd(TagDefinition definition, GameObject source = null)

6.2. TagSub
- 정확한 시그니처:
-- public bool TagSub(TagDefinition definition, GameObject source = null)

6.3. source
- 단계 5의 TraitContext를 위해 지금부터 유지한다.
- 현재 단계에서는 저장하거나 사용하지 않는다.
- Null 또는 명시값 여부가 추가·제거 결과에 영향을 주지 않는다.
- 제거하거나 이름·기본값을 바꾸지 않는다.

7. TagDefinition 유효성

7.1. TagAdd 차단 조건
- definition이 Null
- TagID.None
- TagName이 Null 또는 빈 문자열
- Category == Keyword이며 Trait != Null
- Category != Keyword이며 Trait == Null

7.2. 이름 정책
- Null과 빈 문자열은 차단한다.
- 공백만 있는 이름은 허용한다.
- 앞뒤 공백도 허용한다.
- string.IsNullOrEmpty를 사용할 수 있다.
- string.IsNullOrWhiteSpace로 공백 이름을 차단하지 않는다.

7.3. Category.None
- 금지하지 않는다.
- 특수 분류가 필요 없는 일반 태그로 사용할 수 있다.
- Category.None + Trait 존재는 허용한다.
- Category.None + Trait Null은 비Keyword-Trait Null 규칙으로 차단한다.

7.4. Keyword
- 키워드 여부는 TagCategory.Keyword만으로 판정한다.
- 정상 Keyword 태그에는 Trait이 없어야 한다.
- Keyword + Trait은 유효하지 않은 Definition이다.
- TagAdd 진입 단계에서 차단한다.

7.5. 비Keyword Trait
- Keyword가 아닌 태그에는 Trait이 있어야 한다.
- 비Keyword + Trait Null은 정상 사용에서 나올 수 없는 유효하지 않은 조합이다.
- Error 로그 후 false를 반환한다.

8. TagAdd 처리

8.1. 순서
- definition Null 검사
- TagID.None 검사
- TagName Null/빈 문자열 검사
- Keyword-Trait 검사
- 비Keyword-Trait Null 검사
- 영구 TagInstance 생성
- TryAddInstance 호출
- TryAddInstance 결과 반환

8.2. 정상 추가
- 기존 같은 TagID가 없으면 영구 TagInstance를 생성한다.
- 초기값:
-- Duration = 0f
-- RemainingTime = 0f
-- StackCount = 1
-- IsPerishable = false
- TryAddInstance가 true이면 true를 반환한다.
- Trait 호출 없음
- 이벤트 없음
- 성공 로그 없음

8.3. 유효성 실패
- TagInstance를 생성하지 않는다.
- TryAddInstance를 호출하지 않는다.
- 상태를 변경하지 않는다.
- Debug.LogError 후 false를 반환한다.
- 로그 문구는 영어로 작성한다.
- 직접 로그 인접 위치에 다음 의미의 한글 주석을 작성한다.
-- 개발 단계 12에서 TagDiagnostics로 교체한다.

8.4. 중복 추가 임시 정책
- 같은 TagDefinition이 이미 있으면 false를 반환한다.
- 기존 TagInstance를 유지하고 새 Instance를 등록하지 않는다.
- 로그를 남기지 않는다.
- StackPolicy 값과 관계없이 현재 단계에서는 모든 동일 Definition 중복 추가를 실패 처리한다.
- StackCount를 증가시키지 않는다.
- Refresh 또는 MaxDuration을 적용하지 않는다.
- 이 동작은 초기 CRUD 검증을 위한 임시 구현이다.
- 이후 다음 단계에서 의도적으로 수정한다.
-- 개발 단계 7: 소멸 태그의 영구 전환
-- 개발 단계 10: 영구 StackCount 중복 추가
- 중복 처리 분기 인접 위치에 다음 의미의 한글 주석을 작성한다.
-- 현재 단계에서는 모든 중복 영구 태그 추가를 실패 처리하며, StackCount 정책은 개발 단계 10에서 확장한다.
- 완료 보고에도 임시 정책과 수정 예정 단계를 기록한다.

8.5. 같은 ID의 다른 Definition
- TryAddInstance의 정책을 유지한다.
- 데이터 오류로 처리한다.
- Error 로그 후 false
- 기존 Instance 유지
- 새 Instance 미등록

9. TagSub 처리

9.1. 입력 검증
- definition Null이면 Error 로그 후 false
- TagID.None이면 Error 로그 후 false
- 다음은 제거 차단 조건이 아니다.
-- 빈 또는 공백 이름
-- Keyword-Trait 불일치
-- 비Keyword-Trait Null
-- Category.None
- 잘못된 기존 데이터를 정리할 수 있어야 한다.

9.2. 순서
- definition Null 검사
- TagID.None 검사
- 내부 Dictionary에서 같은 ID 조회
- 없으면 false
- 기존 Instance.Definition과 전달 definition 참조 비교
- 다르면 Error 로그 후 false
- Dictionary에서 해당 ID 제거
- 제거 성공 시 true

9.3. 정상 제거
- 현재 단계에서는 항상 완전히 제거한다.
- StackPolicy를 적용하지 않는다.
- StackCount 감소 없음
- Trait.OnRemove 없음
- 이벤트 없음
- 성공 로그 없음
- 제거 후 TagCheck false
- 제거 후 TryGetTag false와 Null

9.4. 없는 태그
- 로그 없이 false
- 다른 태그에 영향 없음

9.5. 같은 ID의 다른 Definition
- Error 로그 후 false
- 기존 Instance를 제거하지 않는다.
- 상태 변경 없음

9.6. 비정상 기존 데이터
- 다음 데이터가 내부 삽입으로 이미 존재하면 TagSub로 제거를 허용한다.
-- Keyword + Trait
-- 비Keyword + Trait Null
-- 빈 이름
- 현재 단계에는 Trait 생명주기가 없으므로 Trait을 호출하지 않는다.

10. 제거 구현 방식
- public 또는 internal 제거 helper를 새로 만들지 않는다.
- TagSub 내부에서 Dictionary.TryGetValue와 Dictionary.Remove를 사용한다.
- Dictionary 접근 제한자를 변경하지 않는다.
- 단계 4·5에서 이벤트와 Trait 순서가 추가될 예정이므로 제거 helper 구조를 조기 고정하지 않는다.

11. 로그 정책

11.1. Error 로그
- TagAdd Null Definition
- TagAdd TagID.None
- TagAdd 빈 TagName
- TagAdd Keyword + Trait
- TagAdd 비Keyword + Trait Null
- 같은 ID 다른 Definition TagAdd
- TagSub Null Definition
- TagSub TagID.None
- 같은 ID 다른 Definition TagSub

11.2. 로그 없는 실패
- 같은 Definition 중복 TagAdd
- 보유하지 않은 태그 TagSub

11.3. 공통
- 로그 문구는 영어
- 직접 Debug.LogError에는 단계 12 교체 주석
- 성공 로그 없음
